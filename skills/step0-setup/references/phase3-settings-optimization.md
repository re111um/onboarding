# Phase 3: 나만의 Claude Code 세팅

## EXPLAIN

### Claude Code는 "세팅이 반"입니다

Phase 2에서 Output Style, 상태바, 모드를 배웠죠?
사실 이것들은 Claude Code 설정의 **겉면**이에요. 진짜 중요한 건 **안쪽 설정**입니다.

왜 중요할까요? Claude Code는 매 세션마다 "시스템 프롬프트"라는 걸 읽어요.
여기에 **여러분의 설정, 플러그인, 프로젝트 지침**이 전부 들어갑니다.
이 시스템 프롬프트가 크면 클수록 — Claude가 생각할 공간이 줄어들고, 토큰(= 비용)이 더 나가요.

비유하자면: 책상 위에 서류를 잔뜩 쌓아두면 일할 공간이 없어지는 것과 같아요.
**필요한 것만 책상 위에 올려두는 게 중요합니다.**

### 시스템 프롬프트를 구성하는 3가지

| 구성 요소 | 무엇인가 | 토큰 영향 |
|-----------|----------|-----------|
| **CLAUDE.md** | 프로젝트별 지침 (매 세션마다 로드) | 중간 |
| **플러그인** | 스킬, 에이전트, 훅 정의 | **큼** — 가장 많은 토큰 차지 |
| **MCP 도구** | 외부 도구(Notion, Slack 등) 연결 | 작음 (이름만 로드, 사용 시 상세 로드) |

### 핵심 설정 파일 2가지

**1. CLAUDE.md — "이 프로젝트에서는 이렇게 해줘"**

프로젝트 폴더에 `CLAUDE.md` 파일이 있으면, Claude가 **매 세션 시작할 때마다** 자동으로 읽어요.
여기에 적힌 내용을 Claude가 지침으로 따릅니다.

예시:
- "항상 한국어로 답변해줘"
- "커밋 메시지는 한글로 작성해"
- "테스트 없이 코드를 머지하지 마"

**2. settings.json — "Claude Code 자체를 이렇게 동작해"**

`~/.claude/settings.json`은 Claude Code의 핵심 설정 파일이에요.
여기에 담기는 것들:

| 설정 | 역할 | 예시 |
|------|------|------|
| `permissions` | 어떤 도구를 자동 허용/거부할지 | `"allow": ["Bash(git *)"]` |
| `hooks` | 특정 이벤트에 자동 실행되는 스크립트 | 파일 저장 시 린트 실행 |
| `enabledPlugins` | 어떤 플러그인을 켤지/끌지 | 안 쓰는 개발 도구 끄기 |
| `model` | 기본 사용 모델 | `claude-opus-4-6` |

### 모델 설정 — Sonnet vs Opus

설치 가이드에서 모델을 **Opus**로 변경했죠? 그 이유를 좀 더 설명할게요.

| 모델 | 특징 | 비용 |
|------|------|------|
| **Sonnet** (기본) | 빠르고 가벼움. 단순 작업에 적합 | 낮음 |
| **Opus** (권장) | 더 깊이 있는 분석, 복잡한 맥락 이해, 풍부한 교육적 설명 | 높음 |

{{COMPANY_NAME}}은 **팀 플랜**을 사용하고 있어서 Opus 사용에 추가 과금이 없어요.
같은 비용이라면 당연히 더 좋은 모델을 쓰는 게 유리하겠죠?

settings.json에서 확인할 수 있어요:
```json
"model": "claude-opus-4-6"
```

> 💡 빠른 응답이 필요할 때는 Claude Code 안에서 `/model`로 일시적으로 Sonnet으로 전환할 수도 있어요.

### 기본 모드 설정 — 매 세션 Auto mode로 시작하기

설치 가이드에서 기본 모드도 auto로 설정했어요. 이건 settings.json의 이 부분이에요:
```json
"permissions": {
  "defaultMode": "auto"
}
```

이 설정이 없으면 **매 세션마다 `Shift+Tab`으로 Auto mode를 직접 전환**해야 해요.
설정해두면 Claude Code를 시작할 때마다 자동으로 Auto mode로 들어갑니다.

### 권한 모드 — Claude가 얼마나 자유롭게 행동하는가

Phase 2에서 모드 전환(`Shift+Tab`)을 배웠죠? 이건 단순히 편의 기능이 아니에요.
**권한 모드는 Claude가 할 수 있는 행동의 범위를 결정합니다.**

| 모드 | 동작 | 언제 쓰면 좋은가 |
|------|------|-----------------|
| **Ask permissions** (기본) | 매번 허락을 물어봄 | 처음 배울 때, 민감한 작업할 때 |
| **Auto accept edits** | 파일 읽기/편집은 자동, 명령어만 물어봄 | 일반적인 업무 |
| **Auto mode** | 모든 작업을 자동으로 실행 | 온보딩, 신뢰하는 작업 |

왜 이게 중요한지, 실제 상황을 볼게요:

> 🔒 **"설치하지 않는 것을 권장합니다"**
>
> 회사에서 공식적으로 사용하는 사내 도구를 설치하려고 했는데,
> Claude가 "비공식 프로젝트입니다", "외부 서버로 데이터를 전송합니다"라며 **설치를 거부**한 사례가 실제로 있었어요.
> "괜찮아, 설치해줘"라고 해도 — "죄송하지만 설치하지 않겠습니다"라고 다시 거부했습니다.
>
> Claude는 스크립트의 패턴만 보고 위험하다고 판단한 거예요.
> **맥락(이건 우리 회사 도구야)을 모르니까** 기계적으로 차단한 겁니다.

이런 상황에서 할 수 있는 것:
- **Auto mode**라면 Claude가 사용자의 판단을 신뢰하고 진행합니다
- **permissions 설정**에서 특정 명령어를 미리 허용해둘 수 있어요 (`"allow": ["Bash(git *)"]`)
- **CLAUDE.md**에 "이 프로젝트의 스크립트는 신뢰할 수 있다"고 명시할 수도 있어요

> 처음에는 Ask permissions로 시작해서 Claude의 동작을 관찰하고,
> 익숙해지면 Auto mode로 전환하는 것을 추천해요!

### 플러그인이 토큰에 미치는 영향

Claude Code 마켓플레이스에는 수십 개의 플러그인이 있어요.
기본으로 많이 켜져 있는데, **쓰지 않는 플러그인도 시스템 프롬프트에 올라갑니다.**

예를 들어:
- `plugin-dev` (플러그인 개발 도구) → 스킬 7개 + 에이전트 3개 = **10개 항목**이 매 세션 로드
- `pr-review-toolkit` (PR 리뷰) → 에이전트 6개가 매 세션 로드
- 개발자가 아니라면? → **쓰지 않는데 토큰만 소모**

> 안 쓰는 플러그인을 끄면, 같은 비용으로 Claude가 더 깊이 생각할 수 있어요.

## EXECUTE

### 여러분이 할 일

아래 4가지를 직접 해보세요!

**1. 🔒 보안 규칙 적용하기 (필수)**

CLAUDE.md가 뭔지 배웠으니, 가장 중요한 보안 규칙부터 적용합니다.
아래 명령어를 **터미널에 한 번에 복사 → 붙여넣기** 하세요:

```bash
mkdir -p ~/.claude && cat >> ~/.claude/CLAUDE.md << 'SECURITY_RULES'

# GitHub 보안 규칙
- .env, .env.* 파일은 절대 git에 커밋하지 마라
- API 키, 비밀번호, 토큰, 인증서 등 민감 정보를 코드나 파일에 포함하지 마라
- *.key, *.pem, credentials/, secrets/ 등 인증 관련 파일은 절대 커밋하지 마라
- 개인정보(이름, 연락처, 주민번호 등)가 포함된 파일은 절대 커밋하지 마라
- git push 전에 민감 정보가 포함되지 않았는지 반드시 확인하라
- 실수로 민감 파일을 커밋했다면 즉시 경고하고, 해당 키/비밀번호를 변경하도록 안내하라
- git push --force 는 절대 사용하지 마라
SECURITY_RULES
```

> ✅ 최초 **한 번만 실행하면 됩니다.** 기존 설정에 영향 없이, 보안 규칙이 뒤에 추가됩니다.
> 이 규칙이 적용되면 Claude가 API 키나 개인정보를 실수로 GitHub에 올리는 것을 자동으로 방지해요.

**2. CLAUDE.md 확인하기**

방금 추가한 보안 규칙이 잘 들어갔는지, 이 프로젝트의 CLAUDE.md와 함께 확인해보세요:
```
~/.claude/CLAUDE.md 파일 내용 보여줘
```
- 맨 아래에 "GitHub 보안 규칙"이 추가되었는지 확인하세요
- 이 글로벌 CLAUDE.md는 **어떤 프로젝트에서든** Claude에게 매번 전달됩니다
- 프로젝트 폴더의 CLAUDE.md도 확인해보세요: `CLAUDE.md 파일 내용 보여줘`

**3. 현재 플러그인 목록 확인하기**

아래 명령어를 입력해보세요:
```
/plugins
```
- 켜져 있는(enabled) 플러그인 목록을 확인하세요
- 각 플러그인 이름과 설명을 훑어보세요
- "이건 내 업무에 필요 없겠다" 싶은 게 있는지 생각해보세요

**4. 플러그인 정리하기**

확인한 목록 중 내 업무에 불필요한 플러그인을 `/plugins`에서 꺼보세요.

참고 — 직무별 권장 설정:

| 직무 | 유지 권장 | 끄기 권장 |
|------|-----------|-----------|
| **모든 직무 공통** | superpowers, commit-commands, session-wrap, hookify, claude-md-management | LSP 계열 (pyright, typescript 등), agent-sdk-dev |
| **비개발 직무** (HR, PM, 마케팅 등) | skill-creator, context7 | plugin-dev, pr-review-toolkit, feature-dev, frontend-design, code-review, code-simplifier, playground |
| **개발 직무** | feature-dev, pr-review-toolkit, code-review | (대부분 유지) |

> 끈 플러그인은 언제든 `/plugins`에서 다시 켤 수 있어요. 부담 없이 정리하세요!

4가지를 다 해봤으면 "완료" 또는 "다음"이라고 입력해주세요!
